# dear-blog 部署使用说明

更新时间：2026-03-09
默认方案：方案 2（镜像仓库发布）
备用方案：方案 1（手动上传 JAR）

## 0. 文件说明

- `docker-compose.yml`：主编排（默认拉远程后端镜像）
- `Dockerfile.backend`：本地/CI 构建后端镜像使用
- `Dockerfile.runtime`：备用方案用本地 `app.jar` 构建运行镜像
- `docker-compose.manual.yml`：备用方案覆盖文件
- `nginx/nginx.conf`：宿主机 Nginx 站点配置模板
- `.env.example`：环境变量模板

## 1. 前置条件

1. 服务器已安装 Docker、Docker Compose、Nginx
2. 已有站点目录：`/var/www/dear-blog/portal`、`/var/www/dear-blog/admin`
3. 在服务器 `deploy/` 目录下准备 `.env`

```bash
cd deploy
cp .env.example .env
```

至少配置：

- `DB_PASS`
- `REDIS_PASS`
- `JWT_SECRET`
- `BACKEND_PORT`（可选，默认 18080）
- `BACKEND_IMAGE`（方案 2 必填）

## 2. 前端打包与发布（详细）

本节同时适用于方案 1 和方案 2，因为两种方案前端都走静态文件发布。

### 2.1 Portal 打包

在项目根目录执行：

```bash
cd frontend/portal
pnpm install --frozen-lockfile
pnpm build
```

Portal 产物目录：

- `frontend/portal/.vitepress/dist`

### 2.2 Admin 打包

在项目根目录执行：

```bash
cd frontend/admin
pnpm install --frozen-lockfile
VITE_BASE_URL=/admin/ pnpm build
```

如果你在 Windows PowerShell 执行，请改为：

```powershell
$env:VITE_BASE_URL='/admin/'; pnpm build
```

Admin 产物目录：

- `frontend/admin/dist`

说明：

- `VITE_BASE_URL` 必须是 `/admin/`，否则后台资源路径会错。

### 2.3 上传前端产物到服务器

先在服务器创建目录：

```bash
mkdir -p /var/www/dear-blog/portal /var/www/dear-blog/admin
```

示例（Linux/macOS 本地）

```bash
rsync -av --delete frontend/portal/.vitepress/dist/ user@<server>:/var/www/dear-blog/portal/
rsync -av --delete frontend/admin/dist/ user@<server>:/var/www/dear-blog/admin/
```

示例（仅 scp）

```bash
scp -r frontend/portal/.vitepress/dist/* user@<server>:/var/www/dear-blog/portal/
scp -r frontend/admin/dist/* user@<server>:/var/www/dear-blog/admin/
```

### 2.4 Nginx 对应关系

`deploy/nginx/nginx.conf` 中关键映射：

- `root /var/www/dear-blog/portal;` 对应 Portal
- `location ^~ /admin/ { alias /var/www/dear-blog/admin/; ... }` 对应 Admin
- `/api/`、`/uploads/` 反代后端 `127.0.0.1:18080`

部署后执行：

```bash
nginx -t
systemctl reload nginx
```

### 2.5 前端发布验证

- `https://<domain>/` 能打开 Portal
- `https://<domain>/admin/` 能打开 Admin
- 浏览器 Network 中 Admin 静态资源路径是 `/admin/assets/...`
- `https://<domain>/api/...` 有后端响应

## 3. 方案 2（默认）：镜像仓库发布

### 3.1 本地/CI 构建并推送后端镜像

```bash
docker build -f deploy/Dockerfile.backend -t ghcr.io/<org>/dear-blog-backend:<tag> .
docker push ghcr.io/<org>/dear-blog-backend:<tag>
```

建议 `<tag>` 用不可变版本，如 `20260309-01` 或 Git SHA。

### 3.2 服务器拉取并发布

修改 `deploy/.env`：

```env
BACKEND_IMAGE=ghcr.io/<org>/dear-blog-backend:<tag>
```

执行：

```bash
cd deploy
docker login ghcr.io
docker compose --env-file .env pull backend
docker compose --env-file .env up -d
```

### 3.3 回滚

把 `BACKEND_IMAGE` 改回旧 tag 后再次执行：

```bash
docker compose --env-file .env pull backend
docker compose --env-file .env up -d
```

## 4. 方案 1（备用）：手动上传 JAR

适用：镜像仓库不可用或 CI 故障。

### 4.1 本地构建 JAR

```bash
mvn -f backend/pom.xml clean package -DskipTests
```

产物：`backend/blog-app/target/blog-app-*.jar`

### 4.2 上传到服务器 deploy 目录

上传这些文件到同一目录（如 `/opt/dear-blog/deploy`）：

1. `docker-compose.yml`
2. `docker-compose.manual.yml`
3. `Dockerfile.runtime`
4. `.env`
5. `app.jar`（由 `blog-app-*.jar` 重命名）

### 4.3 启动备用方案

```bash
cd /opt/dear-blog/deploy
docker compose --env-file .env -f docker-compose.yml -f docker-compose.manual.yml up -d --build
```

## 5. 常用运维命令

```bash
# 状态
docker compose --env-file .env ps

# 后端日志
docker compose --env-file .env logs -f backend

# 重启后端
docker compose --env-file .env restart backend

# 停止
docker compose --env-file .env down
```
